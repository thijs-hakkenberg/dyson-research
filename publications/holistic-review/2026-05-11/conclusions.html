What's not working

Most significant gaps

  • AI consensus is not validation. Three LLMs trained on overlapping corpora agreeing is weak evidence; their disagreements are arguably more informative. all
  • No integrated reference architecture. Component studies don't yet close into a coupled system showing Phase 0 → 1 → 2 under consistent assumptions. GPT-5.5
  • Cost figures lack uncertainty discipline. Three-LLM votes don't yield uncertainty intervals; downstream BOM treats consensus as point estimates. ClaudeGPT-5.5
  • Microgravity metallurgy is treated as a research question, not a possible show-stopper. 8 orders of magnitude (100 g lab → 50,000 t/yr) is a paradigm gap, not a maturation gap. Claude
  • Circular validation risk. If LLMs generated both the BOM specs and the simulations validating them, the Tier-1 papers will not survive serious peer review. Gemini
  • Reproducibility decays with model deprecation. claude-opus-4-5, gemini-3-pro, gpt-5-2 won't be queryable in 18 months; raw API exchanges aren't archived immutably. ClaudeGPT-5.5
  • "3/3 Accept" terminology and naming inconsistencies. Reads as peer review when it means LLM agreement; paper numbering and model names disagree across documents. GPT-5.5
  • Governance and certification under-prioritized for a multi-century, multi-actor infrastructure project. ClaudeGPT-5.5

Prioritized recommendations

Synthesized from all three reviews; ordered by impact-per-effort.

1
Build a versioned reference architecture (v0.1). One baseline tying mass, power, cost, ISRU flows, manufacturing throughput, and orbit assumptions together. Every paper cites it. Without this, local conclusions may not survive integration. First step: publish "Project Dyson Reference Architecture v0.1" with assumption IDs.
2
Archive raw model exchanges immutably — now. Every consensus document gets an accompanying .raw.json with full prompt, response, model version, timestamp, API metadata. Content-addressed (Zenodo / git-LFS / IPFS). First step: modify scripts/query-bom-specs.js to write raw API responses alongside markdown, retroactively where possible.
3
Run a calibration study for the consensus method before submitting Paper 03. 10–20 engineering questions with known answers (textbook trade studies, declassified NASA decisions). Report agreement-vs-correctness curves. First step: draft a 20-question calibration set this month.
4
Build a claim-evidence register. For every major claim, label the source: empirical, literature-derived, simulated, expert-estimated, AI-generated, speculative. Prevents LLM consensus from being conflated with physical validation. First step: classify the top 50 claims from Papers 01–06.
5
Convert decision gates into quantitative test cards. Pass/fail metrics, test plans, cost, schedule, fallback options for Gates 1 (metallurgy), 2 (cryogenics), 3 (water extraction). First step: write a one-page Gate 2 cryogenic architecture card — it's the nearest gate.
6
Decouple cost figures from "consensus" rhetoric. Replace point estimates in bom-data.ts with explicit ranges; flag items with high inter-model divergence. Add costLow, costHigh, divergenceFlag fields. First step: schema migration in src/lib/types/index.ts.
7
Commission a fallback-architecture paper for the metallurgy show-stopper. Currently a sub-bullet in Paper 04; should be its own analysis (rotational stations, lunar surface, hybrid Earth-launched high-purity components). First step: add as Paper 04b with a 4–6 week effort estimate, before Paper 04's literature review begins.
8
Recruit external expert red-team reviews for the strongest papers before submission. One reviewer each: ISRU economics, swarm/distributed systems, cryogenics, materials processing, AI methodology. First step: draft a 1-page expert-review request and send it in the next 30 days.
9
Reframe the AI consensus method as hypothesis generation, not validation. Rewrite Paper 03's framing; structured ideation, critique, and uncertainty mapping — not engineering proof. First step: revise the abstract and introduction; demote consensus-as-evidence claims.
10
Promote governance (Paper 09) up the priority order. For a multi-century non-profit aiming to build infrastructure outliving any signatory, governance is a technical requirement, not an afterthought. Find a co-author in space policy. First step: outreach to one or two named space-policy academics.

Bottom line

A real intellectual core inside a frame that's still maturing.

The methodology paper, the ISRU crossover paper, and the risk-gated framework all stand on their own. The Dyson-swarm framing is doing useful work as a forcing function for these contributions, but the corpus currently overstates how close any of this is to a construction plan.

The next six months should produce one integrated architecture, one calibrated methodology, one immutable archive, and explicit repositioning as a feasibility-and-validation program. With that, Paper 03 has a real shot at being cited well outside this project's gravity well.
