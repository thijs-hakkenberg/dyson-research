## 1. Significance & Novelty — **Rating: 4 (Good)**

The manuscript addresses a genuinely important systems-engineering question: what coordination architectures remain viable as autonomous space swarms scale into the \(10^3\)–\(10^5\) (and aspirationally \(10^6\)) regime under a fixed per-node control-plane budget. The paper’s central value is not the asymptotic result (hierarchical \(\eta = O(1)\) is, as you correctly note, analytically implied by an \(O(N)\) message structure), but the *engineering envelope* view: byte-level accounting under explicit workload profiles, coordinator ingress stress-testing, and a clear separation of “baseline telemetry” vs “protocol overhead.” For T-AES readership, the design-parameter framing (cluster size, duty cycle, coordinator capacity, link loss models) is a strong fit.

Novelty is moderate-to-good. Discrete-event simulations of constellation networking exist, and hierarchical coordination is a well-known pattern; however, the combination of (i) explicit byte accounting tied to a per-node budget, (ii) workload-dependent overhead envelope (stress/event/nominal) presented as an architect’s interpolation tool, (iii) coordinator offered-load thresholds (unscheduled vs TDMA), and (iv) AoI used as a coordination-quality proxy, is a useful synthesis that is not commonly presented in the space-swarm coordination literature. The addition of a sectorized mesh comparator helps avoid a strawman “global mesh” baseline and makes the comparison more credible.

That said, the paper sometimes overstates “to our knowledge no prior work…” (Introduction) without sufficiently narrowing the claim. There is substantial related work in mega-constellation routing/control-plane scaling, DTN/disruption-tolerant control, and hierarchical/clustered control in UAV swarms and sensor networks that could partially overlap. The contribution remains valuable, but the novelty claim should be tightened to the specific combination of assumptions and metrics (byte-level overhead under a strict per-node control-plane budget + DES + workload envelope + coordinator offered-load bounds).

## 2. Methodological Soundness — **Rating: 3 (Adequate)**

The cycle-aggregated DES is appropriate for the research questions as posed—particularly for throughput/overhead accounting and queue/buffer stress tests—because packet-level dynamics are not the focus. The manuscript is careful to describe what is modeled vs abstracted (Table “Simulation Abstraction Scope”), and it correctly warns that the reported overhead is “message-layer offered load,” not a full link budget. The analytical cross-check (Eq. (analytical_crosscheck) and Table “Hierarchical Communication Overhead Scaling”) is a good implementation validation step.

However, several modeling choices materially drive the quantitative outputs, and some are currently under-justified or internally inconsistent:

* **Coordinator queueing/latency mechanism**: You attribute the dominant \(\sim 500\) ms latency at the regional coordinator to “burst arrivals near the end of each cycle” (Results, cluster-size discussion). Yet earlier you state nodes have uniformly random phase offsets within the cycle, which should *reduce* burstiness at the cluster coordinator; for the regional coordinator, cluster summaries are generated by cluster coordinators—are those summaries phase-jittered, or are they synchronized at cycle boundaries? If synchronized, the “burst” explanation is plausible; if not, the reported discrete jumps (Table “Hierarchical Protocol Overhead and Latency vs Cluster Size”) need a more explicit event schedule description. Right now the latency story depends on an implicit synchronization assumption that should be stated and sensitivity-tested.

* **Traffic model realism**: The stress-case assumes *one unique 512 B command per node per cycle* (10 s). That is intentionally conservative, but it dominates \(\eta\) and makes the “hierarchical overhead” mostly a *workload assumption* rather than an architectural consequence. You do mitigate this with nominal/event-driven profiles (good), but the paper’s headline \(\eta \approx 46\%\) is still heavily determined by a command model that may not map to many real autonomy concepts (often commands are per-cluster, per-task, or policy updates). A stronger methodology would parameterize command entropy (unique vs broadcast vs compressed deltas) and show \(\eta\) as a function of command structure, not only rate.

* **Link model vs retransmission**: Retransmission is modeled as additional propagation delays only (Link Availability Sensitivity), and offered-load is approximated rather than directly counted from DES attempts (“Offered” column note). Since overhead is a central metric, offered-load should be *measured* from simulated retransmission attempts, not approximated with a formula, especially when later conclusions hinge on infeasibility at \(p_{\text{link}}\le 0.5\). Similarly, Gilbert–Elliott transitions “once per cycle” is a coarse discretization; if cycle is 10 s and mean bad-state is 20 s, you only get 2-step bursts—fine for a first cut, but it should be acknowledged as a discretization artifact with sensitivity to \(T_c\).

Reproducibility is promising (code repository promised, parameter tables provided), but the commit hash is “PENDING.” For an IEEE T-AES submission, the paper should either (i) provide the exact commit hash and a release tag, or (ii) remove the claim of availability until it is concrete.

## 3. Validity & Logic — **Rating: 3 (Adequate)**

Many conclusions are supported within the paper’s abstraction: hierarchical overhead ratio constancy, sectorized mesh higher constant factor due to heartbeats, retransmission ineffectiveness under correlated outages, and coordinator ingress thresholds under unscheduled vs TDMA. The manuscript is generally careful about interpreting baselines as bounds (Baseline Interpretation Note) and about distinguishing analytical inevitability from DES validation (Scaling Behavior section). The limitations section is unusually candid for this genre, which strengthens credibility.

The main concern is that some “engineering” conclusions are presented with more specificity than the model supports:

* **Coordinator capacity thresholds (50 kbps unscheduled / 24 kbps TDMA)**: These are sensitive to (a) within-cycle traffic shaping, (b) whether the coordinator must ingest only ephemeris or also ACK/heartbeat/collision alerts, (c) whether commands share the same terminal/channel, and (d) half-duplex/pointing/acquisition losses (explicitly abstracted). You do label them “offered-load lower bounds,” but the text sometimes reads as if these are directly actionable hardware requirements. Consider presenting them as *functions* \(C_{\text{coord}}(k_c, r, S_{\text{eph}}, \gamma)\) with a clear statement of included/excluded traffic directions and terminal sharing assumptions, and avoid single-number emphasis.

* **AoI interpretation**: The AoI results are consistent with geometric inter-report times under Bernoulli exception reporting, but the operational interpretation (“conjunction screening timelines”) is not fully grounded. AoI at the coordinator is not equivalent to *risk* unless coupled to orbital prediction uncertainty growth. You acknowledge this, but the paper still uses AoI percentiles as if they directly map to screening sufficiency. Tighten language: AoI is a proxy for freshness, not a direct safety metric.

* **Sectorized mesh model**: The “sector coordinator” and status-to-coordinator element makes the comparator partly hierarchical. This is not necessarily wrong (many “decentralized” systems become hybrid in practice), but it weakens claims about “peer-to-peer vs hierarchy” unless framed explicitly as “hybrid gossip + local coordinator.” Also, the choice \(k_s=\lceil\sqrt{N}\rceil\) is asserted as modeling a screening volume; this needs justification from orbital geometry/density (e.g., expected number of neighbors within a screening radius in a shell), otherwise it appears chosen to produce a convenient intermediate scaling law.

Overall, the logic is consistent, but several quantitative claims should be reframed as *conditional on the message model* and key parameters. The paper is already close to that stance; it just needs tighter wording in the abstract/conclusion and in the coordinator capacity sections.

## 4. Clarity & Structure — **Rating: 4 (Good)**

The manuscript is well organized, with clear research questions, explicit baselines, and a strong “what is in/out of scope” table. The definitions of \(\eta\), baseline telemetry vs overhead, and “delivered vs offered” are particularly helpful. Figures appear to be designed to tell the scaling story (overhead vs N, latency distributions, sensitivity sweeps, workload envelope), and the text does a good job walking the reader through why curves look as they do.

Two clarity issues are worth addressing:

1. **Terminology overload / metric consistency**: You use \(\eta\), \(\eta_{\text{total}}\), delivered \(\eta\), offered load, and \(\eta_{\text{eff}}=\eta/\gamma\). This is correct but cognitively heavy. Consider a single “capacity accounting” diagram early (perhaps in the overhead definition section) that shows: physical rate → MAC efficiency → available control-plane budget → baseline telemetry → protocol overhead → retransmission headroom. Also, ensure that every figure/table clearly states whether it is message-layer, MAC-adjusted, delivered, or offered.

2. **Latency reporting units and path definition**: Table “Hierarchical Protocol Overhead and Latency vs Cluster Size” reports “Latency (ms)” but the narrative later decomposes latency for node→cluster coordinator and regional queueing. It is not always clear whether the table’s latency is node→cluster, node→regional, or end-to-end (node→ground). Define the latency path unambiguously in the table caption and in the metric definitions section (you define “message latency” generally, but each result table should pin down the path).

The abstract is dense but largely accurate. It may be slightly overpacked with numbers and claims (e.g., coordinator thresholds, AoI P99, sectorized ratio, GE implication) for T-AES style; consider trimming to 2–3 headline results and moving some specifics into the introduction/contributions.

## 5. Ethical Compliance — **Rating: 4 (Good)**

The manuscript includes an explicit disclosure of AI-assisted ideation in the Acknowledgment, and it distinguishes that ideation from validated results (“not validated in the current study”). That is good practice and aligns with emerging IEEE expectations on transparency. There is no apparent human-subjects component, dual-use risk is minimal at this abstraction level, and the work is primarily simulation-based systems analysis.

Two items to improve:

* **Conflict-of-interest / affiliation clarity**: The author block is “Project Dyson Research Team” with deferred names/affiliations. IEEE policy may allow blinded or placeholder authors during review, but final transparency matters. Since the work references Project Dyson and provides a repository under that organization, include a brief statement in the final version clarifying governance/funding and any commercial ties (if any), especially if the results may be used to justify architectural decisions.

* **AI tools and reproducibility**: Since AI systems are named, consider adding one sentence clarifying that AI tools were not used to generate simulation code/results, only ideation text. This prevents reviewer concern about provenance of numerical results.

## 6. Scope & Referencing — **Rating: 3 (Adequate)**

The topic is appropriate for IEEE Transactions on Aerospace and Electronic Systems: constellation/swarm coordination architectures, offered-load sizing, resilience under loss, and queueing-informed analysis are all within scope. The paper’s references cover foundational distributed algorithms (Lynch, Lamport, Raft), gossip (Demers), AoI (Kaul/Yates/Sun), and some constellation networking (Handley, del Portillo, Akyildiz). The inclusion of DTN/BPv7 is relevant given the store-and-forward conclusion.

Concerns:

* **Non-archival / web citations**: Several key claims rely on non-archival web sources (Starlink ops page, DARPA program pages, Google S2 library, Project Dyson publication page). For T-AES, it would be better to anchor operational claims in archival sources (FCC filings, peer-reviewed analyses, operator technical papers) where possible. At minimum, ensure that non-archival citations are not used as primary evidence for quantitative statements.

* **Missing related work on hierarchical/cluster-based control planes and constellation management**: There is a large body of work on clustered WSN protocols, hierarchical routing/control (LEACH-like paradigms), and on constellation control-plane design that could strengthen the positioning. Even if not space-specific, citing representative cluster-based communication/control scaling papers would help address novelty skepticism.

* **Akyildiz 2003 for optical ISL/TDMA efficiency**: Some citations feel mismatched to the specific claims (e.g., using a 2003 satellite IP multicast paper to justify modern optical terminal counts and TDMA efficiency). Consider updating with more recent optical ISL / LEO networking measurements and standards work (CCSDS optical comm references, recent LEO ISL architecture papers, etc.).

## Major Issues

1. **Regional coordinator latency “burst” mechanism is underspecified and may be an artifact**  
   The reported \(\sim 340\)–675 ms mean latencies and the explanation that regional queueing dominates due to end-of-cycle bursts need a precise description of *when* cluster summaries are generated and injected (synchronous at cycle boundary vs jittered). If synchronous, justify why; if not, reconcile with the burst claim. Add a sensitivity run where cluster-summary emission is phase-randomized to show whether latency conclusions (and “optimal \(k_c\)” conclusions) persist.

2. **Offered-load with retransmissions should be measured, not approximated**  
   Table “Coordination Success and Overhead vs Link Availability” uses an approximation for offered load with retries. Since link-loss conclusions hinge on infeasibility due to saturation, the DES should directly account attempted bytes (including retries) and report offered load with confidence bounds (even if near-deterministic). This is a substantial but straightforward revision.

3. **Sectorized mesh comparator needs stronger grounding and clearer framing as a hybrid**  
   The choice \(k_s=\lceil\sqrt{N}\rceil\), the role of the “sector coordinator,” and the capped fanout of 10 are key determinants of the 1.4–1.5× overhead factor. Provide justification from orbital density/screening volume arguments (or clearly label them as illustrative), and explicitly frame the comparator as “local coordinator + peer heartbeats” rather than a purely decentralized mesh. Otherwise, readers may view the comparison as parameter-tuned.

4. **Command model drives headline overhead; broaden the command-structure parameterization**  
   With 512 B *unique* per-node commands every 10 s, \(\eta\) is dominated by downlink commands. Add at least one additional command-structure case: (i) per-cluster broadcast command, (ii) delta-encoded commands, or (iii) policy update per \(m\) cycles. This would make the “design envelope” more generally useful and reduce the risk that the main quantitative result is seen as an artifact of an extreme command assumption.

5. **Repository/commit hash must be concrete or removed**  
   “commit hash: [PENDING]” undermines reproducibility claims. For review, you can keep the repository link, but for publication you need a fixed revision identifier and a minimal replication guide (config files, scripts to reproduce key figures).

## Minor Issues

1. **Inconsistency in workload overhead numbers**  
   Abstract says event-driven \(\eta \approx 6\%\) and nominal \(\eta \approx 5\%\). Fig. “workload” caption later states event-driven \(\sim 7\%\) and nominal \(\sim 6\%\). Align these across abstract, contributions, and figure captions.

2. **Equation/notation clarity**  
   * Eq. (mac_efficiency) defines \(C_{\text{raw}}=C_{\text{coord}}/\gamma\), but earlier \(\eta_{\text{eff}}=\eta/\gamma\). Consider using consistent notation for “effective” vs “raw” across both bandwidth and overhead.  
   * In Table “Mesh Traffic Accounting,” “Gossip exchange (mesh)” size is listed as \(256\times f\) in Table “Traffic Accounting,” but earlier it is \(256\times b\) per message with \(f\) messages per round. Ensure consistent representation.

3. **Centralized baseline overhead definition**  
   Table “Topology Comparison” lists centralized overhead “5–15%” as protocol overhead beyond baseline. It’s unclear what messages constitute that overhead in the centralized model (ACKs? commands?). Provide a short traffic accounting analogous to Table “Traffic Accounting” for centralized, or remove the overhead range if it is not computed the same way.

4. **Latency distribution figure includes \(10^6\) analytical extrapolation**  
   Fig. “latency distribution” includes a \(10^6\) curve labeled analytical extrapolation. Consider moving this to an appendix or clearly separating it visually (different line style + “not validated”) to avoid readers over-weighting it.

5. **Failure model statement**  
   “2% annual failure rate → MTTF 50 years per node” is correct for exponential with constant hazard, but readers may question applicability to LEO smallsats where early-life failures dominate. A brief note about “operational phase only” is present; consider adding a sentence about burn-in/bathtub curve and why exponential is used here.

## Overall Recommendation — **Major Revision**

The paper is strong in motivation, framing, and transparency, and it likely will be publishable in T-AES after revision. The main reasons for Major Revision are (i) key quantitative conclusions (latency vs \(k_c\), coordinator capacity thresholds, retransmission feasibility) depend on underspecified within-cycle scheduling and on approximated offered-load accounting, and (ii) the sectorized mesh comparator and command model need stronger justification/parameterization to avoid the impression that constants were chosen to produce the reported ratios. These are fixable with targeted additional simulations, clearer model specification, and modest expansion of the workload space.

## Constructive Suggestions

1. **Add a “within-cycle timing model” subsection and a sensitivity test**  
   Explicitly define when each message type is generated within the cycle (node reports phase-random; are coordinator summaries phase-random or end-of-cycle?). Then add one sensitivity run randomizing coordinator summary emission times to test whether regional queueing (and the \(k_c\) latency trade-off) persists.

2. **Instrument DES to report *offered* bytes directly under retransmission and GE loss**  
   Replace the offered-load approximation in Table “link availability” with measured offered bytes. Add a plot of offered load vs \(p_{\text{link}}\) for \(M_r\in\{0,1,2\}\), and show saturation points including baseline telemetry.

3. **Broaden the command model beyond “unique per-node per-cycle”**  
   Add at least two command structure variants: per-cluster broadcast (1 command per cluster per cycle) and event-driven per-node commands with variable payload size (delta/policy). Report \(\eta\) as a function of “command uniqueness fraction” and “command size,” which will generalize the design envelope.

4. **Justify sector sizing from orbital density / screening radius, or relabel as illustrative**  
   Provide a short derivation mapping a screening volume (e.g., within \(R\) km along-track/cross-track/radial) to expected neighbor count at a given shell density, and show how that compares to \(\sqrt{N}\). If this is not possible within scope, clearly state that \(k_s=\sqrt{N}\) is an illustrative scaling choice and include a sensitivity sweep over \(k_s\) and fanout cap.

5. **Tighten consistency and presentation of headline numbers**  
   Align nominal/event-driven overhead values across abstract/contributions/figures, and add a single summary table that lists (for each workload) total utilization = baseline + protocol, plus MAC-adjusted utilization for a representative \(\gamma\). This will reduce reader confusion and make the “design envelope” easier to apply.
